# RFC 0001 — Tier-3 IR extensions for v1.5

Status: draft \
Author: manysql core \
Targets: v1.5 (post-v1 lock)

## Summary

v1 of manysql ships an IR scoped to ANSI-core read-only relational algebra
plus windows, CTEs (including recursive), set ops, and correlated subqueries.
Tier-1 semantic divergence (null ordering, division-by-zero, identifier
folding, set-op default, etc.) is fully expressed via `SemanticConfig` —
not via IR shape.

This RFC plans the v1.5 IR extensions that bring the system to feature
parity with the *interesting* surface area of modern OLAP dialects:
arrays / structs / JSON, regex flavor selection, and deep date/time
arithmetic. Each extension is opt-in per dialect, behind a `DialectSpec`
feature flag, and accompanied by a verification plan.

The unifying constraint: **every Tier-3 feature must be representable as
new IR nodes or expression nodes that compose with the existing ones.**
None of these extensions are allowed to redesign existing nodes. If a
proposed feature can only be expressed by reshaping `Project` or
`Aggregate`, it is out of scope and belongs in a sibling system.

## Goals

1. Cover the dialect surface area that distinguishes BigQuery, Snowflake,
   Databricks, ClickHouse, and DuckDB from each other and from ANSI core.
2. Keep the IR's flat, dataclass-of-tuples shape — additive only.
3. Keep the executor parameterized: every new node has one handler in
   `manysql/executor/engine.py`; every new expression has one entry in
   `ExprEvaluator`.
4. Each feature's correctness is verifiable by at least one oracle
   (DuckDB, ReferenceInterpreter, or a property oracle) without needing
   a new external engine.
5. Generated dialects can opt into a feature without paying its cost
   when they don't need it (codegen lane is short-circuited by the
   spec's feature flags).

## Non-goals

- DDL or DML extensions (CREATE / INSERT / UPDATE / DELETE / MERGE).
  Those land in a separate RFC if scope ever expands beyond reads.
- Streaming, time-travel storage, federation across catalogs.
- Procedural extensions (PL/SQL, T-SQL stored procs, anonymous blocks).
- Probabilistic queries, sampling. (Sampling is a smaller separate RFC.)

## Phases

We ship Tier-3 in three phases so each lands with end-to-end
verification before the next begins.

### Phase 3A — typed values: arrays, structs, maps

Most modern OLAP dialects expose three composite types. Each varies in
syntax and indexing convention but the IR can absorb the divergence with
a small number of new nodes and a SemanticConfig knob set.

#### IR additions

Expression nodes:

- `ArrayLit(elements, element_type)` — `[1, 2, 3]` / `ARRAY[1, 2, 3]`.
- `StructLit(fields)` — ordered named tuple values.
- `MapLit(entries)` — kv pairs; only added when at least one target
  dialect cannot encode maps as `ARRAY<STRUCT>`.
- `Subscript(container, index, kind)` — array access; `kind` is
  `ARRAY` / `MAP` / `STRUCT`. Index expr type is checked statically.
- `FieldAccess(struct_expr, field_name)` — `s.field`.
- `ContainerLength(expr)` — `ARRAY_LENGTH` / `CARDINALITY` / `MAP_LENGTH`.

Plan nodes:

- `Unnest(input, expr, with_ordinality)` — flatten an array into rows.
  Lowers `LATERAL UNNEST(...)` and friends. The cross product with
  the outer relation is implicit.
- (No `Pivot` node yet; that is its own RFC.)

Type system:

- New `IRType.kind`: `ARRAY`, `STRUCT`, `MAP`. Element / field types live
  in the existing `params` slot.

#### SemanticConfig knobs (Tier 1.5)

- `array_base_index: 0|1` — already in v1; promote to required when
  arrays are enabled.
- `array_null_propagation: PROPAGATE | NULL_AS_EMPTY` — Postgres vs
  BigQuery on `array_length(NULL)`.
- `struct_field_case: CASE_SENSITIVE | CASE_INSENSITIVE`.
- `map_unique_keys: REQUIRE | LAST_WINS | FIRST_WINS`.

#### DialectSpec extension

```
SurfaceSpec.array_literal_syntax: ARRAY_KW | BRACKETS | LIST_KW
SurfaceSpec.struct_literal_syntax: STRUCT_KW | PARENS_NAMED | OBJECT_KW
SurfaceSpec.map_literal_syntax: MAP_KW | KV_PAIRS | UNSUPPORTED
SurfaceSpec.unnest_syntax: LATERAL_UNNEST | TABLE_FN | EXPLODE
```

#### Executor

Polars has native list / struct / object types; arrays + structs map
directly. Maps are emulated via `list<struct<key,value>>` with index
methods on the executor. `Unnest` becomes `explode`.

#### Verification plan

- DuckDB oracle: full coverage for arrays / structs / maps; the SQL
  rewriter emits DuckDB-flavored literal syntax.
- SQLite oracle: skipped via capability metadata
  (`unsupported_nodes={"ArrayLit", "StructLit", "Unnest"}`).
- ReferenceInterpreter: extended with hand-coded list/dict semantics.
  Slow but ours; serves as the ground truth for SemanticConfig knobs
  no SQL engine expresses (e.g., `map_unique_keys=FIRST_WINS`).
- PropertyOracle: `Unnest` row-count = `sum(len(arr) for each input row)`;
  `Subscript` width = 1; `ArrayLit.length` matches input length.

#### Risks

- Polars's nested-type API has evolved across versions; pin a known-good
  version range. Worst case we emulate via PyArrow.
- `MapLit` adds a third composite type to maintain forever. Defer
  unless a target dialect genuinely cannot use `ARRAY<STRUCT>`.

### Phase 3B — JSON

JSON is structurally a typed-value extension (Phase 3A) plus a path
language. We separate it into its own phase because the path language
needs its own lexer/parser and SemanticConfig knobs.

#### IR additions

- `JsonExtract(expr, path_str)` — `expr -> path` in Postgres,
  `JSON_EXTRACT` in MySQL, `expr.path` in BigQuery.
- `JsonExtractText(expr, path_str)` — `->>` variant: result coerces
  to text.
- `JsonContains(haystack, needle)` — `@>` in Postgres, `JSON_CONTAINS`
  in MySQL.
- `JsonArrayLength(expr)` — array-length on a JSON array.
- `JsonType(expr)` — returns the JSON type as text.

Path language: a JSON Path subset (object access, array index, wildcard,
recursive descent). Parsed once, stored as a small AST inside
`JsonExtract`.

#### SemanticConfig knobs

- `json_missing_field: NULL | ERROR` — Postgres returns NULL,
  Snowflake errors on strict mode.
- `json_path_dialect: JSON_PATH | DOT_ACCESS | BRACKET_DUAL` — surface
  syntax flag that filters which path syntaxes the dialect accepts.
- `json_strict_array_index: ZERO_BASED | ONE_BASED`.
- `json_string_escape: STANDARD | EXTENDED`.

#### DialectSpec extension

```
SurfaceSpec.json_extract_op: ARROW | DOT | FN_CALL
SurfaceSpec.json_extract_text_op: DOUBLE_ARROW | TEXT_FN | INFERRED
```

#### Executor

Polars has `pl.Expr.str.json_path_match`. For non-trivial paths we
delegate to a small JSON path evaluator in
`manysql/executor/json_path.py` (~200 lines, pure Python).

#### Verification plan

- DuckDB oracle: full coverage. SQL rewriter emits DuckDB JSON
  operators.
- ReferenceInterpreter: hand-coded path evaluator using the same AST.
- PropertyOracle: `JsonExtract(NULL_JSON, anything) = NULL`;
  `JsonArrayLength(arr) >= 0`; `JsonType` returns one of a fixed
  enumerable set.

### Phase 3C — regex flavor + deep date/time

Two cleanly-separated mini-extensions that can ship together because
they share a common "engine selector" pattern (regex flavor, date format
flavor) and the same lifecycle.

#### Regex flavor

IR node: `RegexMatch(value, pattern, flags)` already implicitly used by
`LIKE` lowering for some dialects. Promote to a first-class IR node so
flavor differences are explicit.

SemanticConfig knobs:

- `regex_flavor: POSIX | PCRE | RE2 | JAVA` — controls which features
  the regex engine accepts (lookahead, possessive quantifiers, etc.).
- `regex_default_flags: str` — e.g. `"i"` or `""`.
- `regex_unicode_class_support: bool`.

Executor: dispatch by `regex_flavor` to either Python `re`, `pcre2`, or
a vendored RE2 binding. Default `POSIX` falls through to Python `re`.

Verification plan:

- ReferenceInterpreter: each flavor selects a different Python library.
  Flavor disagreement is exposed via PropertyOracle (regex must
  agree with itself across runs).
- DuckDB oracle: only valid for the subset of patterns DuckDB accepts.

#### Deep date/time

Tier 1 already covers basic timestamp + date arithmetic. Tier 3 covers:

- `Timestamptz` distinct from `Timestamp` (with offset).
- Calendar interval arithmetic that respects month-end boundaries.
- `AT TIME ZONE` expressions and `EXTRACT` with non-trivial fields
  (`ISO_DOW`, `WEEK`, `QUARTER`, `EPOCH`, `JULIAN`).
- `DATE_TRUNC` with non-stdandard precisions (`century`, `millennium`,
  `quarter`).
- `DATE_DIFF` semantics: BigQuery/Snowflake disagree on whether
  `DATE_DIFF('month', 2024-01-31, 2024-03-01)` is 1 or 2.

IR additions:

- `Extract(field, value)` — already exists; widen `field` enum.
- `DateAdd(value, interval, unit, semantics)` — make the rounding
  policy explicit.
- `DateDiff(a, b, unit, semantics)` — same for diff.

SemanticConfig knobs:

- `date_diff_policy: TRUNCATING | FENCEPOST` — covers the BigQuery
  vs Snowflake disagreement.
- `date_add_month_end_policy: SATURATE | ROLL` — Postgres vs Oracle.
- `interval_supports_year_to_month: bool`.

Verification plan: ReferenceInterpreter is the ground truth for the
new policy knobs; DuckDB and SQLite each cover one of the two policies
natively, so they're capability-flagged.

## Cross-cutting changes

### Spec → grammar codegen

Each phase adds new optional grammar rules behind feature flags:

```
SurfaceSpec.features: set[str] = {"arrays", "structs", "maps", "json", ...}
```

The grammar emitter consults `features`; if a flag is off, the relevant
rules are excluded entirely. This keeps grammars small for dialects
that don't opt in.

### Spec → lowering codegen

Lowering for new IR nodes is *only* generated when the spec's surface
includes the corresponding rule. The deterministic emitter produces a
stub that delegates to `lowering_helpers.lower_array_lit` etc. — these
helpers live in the reference and are reused unchanged unless the spec
declares structural divergence.

### Operator-override scaffold

Phase 3A introduces enough new functions that the operator-override
mechanism (already in place for Tier-1 dialects) becomes load-bearing:
any function the executor doesn't implement directly falls through to
`overrides.FUNCTIONS`. The LLM lane populates these for genuinely novel
dialect features.

## Migration & compatibility

Tier-3 lands in v1.5. Existing v1 dialects continue to work unchanged:
their `DialectSpec.features` is empty by default, and codegen produces
the same artifacts as today.

Tier-3-aware specs declare `features` explicitly. The codegen pipeline
fails fast if a spec asks for `arrays` but its `array_literal_syntax`
is unset.

## Verification budget

Each phase is gated on:

1. The corresponding ReferenceInterpreter extension passing 100+
   golden queries that exercise the new IR nodes.
2. The DuckDB oracle (extended) agreeing with the executor on every
   query that doesn't depend on a Tier-3 SemanticConfig knob outside
   DuckDB's coverage.
3. CrossDialectOracle: at least three generated Tier-3 dialects (e.g.,
   "BigQuery-flavored arrays", "Snowflake-flavored structs",
   "Postgres-flavored JSON") agreeing on a 50-query battery.
4. PropertyOracle: structural invariants extended to cover the new
   nodes. `Unnest` row count, `JsonExtract` null-propagation,
   `RegexMatch` boolean output.

## Open questions

- **Should `MapLit` ship in 3A or wait?** A real cost-benefit decision
  that depends on which target dialects we plan to model first. Default
  recommendation: defer to 3A.1 unless a target dialect literally cannot
  be modeled with `ARRAY<STRUCT<key, value>>`.
- **JSON path AST: ours or jsonpath-rfc9535?** The RFC stabilizes the
  syntax but is heavier than we need. Default recommendation: subset of
  RFC9535, hand-written parser, ~200 LoC.
- **Regex engine pinning.** PCRE2 via `pcre2` Python binding is fast but
  not pure-Python. RE2 via Google bindings is pure but heavier. Default
  recommendation: ship POSIX + Python `re` first; add PCRE/RE2 as an
  optional dependency in 3C.1.
- **Tier-3 oracle for array-of-struct equality.** Set comparison breaks
  when rows contain unhashable composites. Default recommendation:
  recursive structural compare in `frames_equal`, with normalization of
  array element order when the SQL is order-insensitive.

## Out-of-scope (record for posterity)

These belong in their own RFCs:

- Sampling (`Sample` node).
- Pivot/Unpivot.
- Time-travel `Scan` (versioned tables).
- Lateral table-valued function calls beyond `UNNEST`.
- Recursive types (`array<array<int>>` etc. exist but recursive type
  *aliases* do not).
